Version 2 Design history

Goals of this iteration

Why we explored iterating agreements and exploring payments information

Pain points and user needs

We did a lot of research into Farmer's understanding of the action centred agreement concept. This included:

  • interviews to discuss concept of action centred agreement to reduce confusions (too many schemes) and align more with farmers needs (eg ability to change or copy actions quickly)
  • testing the v1 prototype with 10 farmers - adding new actions to a new agreement
  • see report here

From this, we identified:

  1. the overall concept was understood and aligned with addressing pain points identified here
  2. the farmers could use the core functions of the prototype easily
  3. that we could extend the 'Agreements' concept to include more complex scenarios - copying a set of actions and adding them to an existing agreements
  4. that we should understand how payments should work for this scenario, and include this within the 'Agreements' screens

User needs:

  1. "As a Farmer or land manager in an agri-environment scheme I need to be able to apply for additional funding, to bring in more land, undertake more actions or buy more items, while already in an agri-environment scheme so that I can do more than originally planned and respond to changing circumstances." Full set of user needs found here.
  2. I need to know in advance when I will get a payment and how much I will receive
  3. I need to know what my payments are for
  4. I need to see the progress and state of my payment
  5. I need to know how my payments has been calculated and over what period
  6. I need to be told when I need to take action and how
  7. I need to analyse and query historic and current payments data

Constraints

The current payments hub cannot handle multiple actions with different end dates within a single agreement. Our approach to this constraint is:

  • to indentify how users need the payments scheduling information to be dispayed for the above scenario
  • analysis on the various payments scheduling scenarios for an action centred agreement
  • making recommendations on the required changes to the payment hub which are needed based on our analysis, research and user testing of these designs
  • to work with the payments hub and the Single Front Door team to look at how this change is impacted into the payments schedule and reflected on the customer facing payment statement.

Design approach

For payments, we've aligned with approach used within the Single Front Door team (see their design history on payments) as stated below.

"The UCD team was directed not to be constrained with the initial design exploration. This meant ideating as if there are no operational or technical blockers. The only focus of our design would be to meet user needs - we needed to imagine an ideal world. For example, designing with zero constraints means we could imagine simplifying and standardising complex payment information across multiple schemes, and farmers getting accurate, timely, up to date payment notifications and schedules."

How might we statements

  1. How might we allow users to quickly amend actions undertaken without the need for a full new application?
  2. How might we allow users to add an amendment to an existing agreement?
  3. How might we allow users to view this completed amendment within an existing agreement?
  4. How might we make users aware of when and how much they will be paid? (from SFD)
  5. How might we make it clear what each payment is for? (from SFD)
