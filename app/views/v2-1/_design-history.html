Version 2.1 Design blog

Background

The previous prototype focused on adding new actions to an existing agreement (version 2). It was decided, following a the 2nd round of User Research with farmers, that we were confident in this area for the alpha phase.

One part of the journey we hadn't deeply explored was how to show an updated agreement so a user can decide whether to accept or reject it.

In this scenario:

  1. a user has applied to add new actions to an existing agreement
  2. the Rural Payments Agency (RPA) has accepted the updates and contracted the user to either accept or reject the update

We put together a (very) basic screen flow (opens in a new tab), so we were all on the same page.

Design approach

The problem to consider was how to show the user the updated version of the agreement with both the existing actions as well as the new actions, and any resulting changes in payment, annual declaration and other legal information.

We then did some exploration; considering the things we may want to learn from research, looking for other services that have done a similar thing, as well as looking for patterns and ways the information could be displayed.

We came up with some ‘how might we’ and ‘we could…’ statements to help us ideate a solution.

‘How might we’ (HMW) statements

  1. HMW make it easy for users to see the latest change they've signed up for as part of the work they're doing on their land?
  2. HMW make it easy for users to see how the change will affect their monthly payments?
  3. HMW make it easy for users to accept their updated agreement?

Ideation

The ideation work was carried out on the FRPS UCD mural board (opens in new tab).

We played around with some ‘we could...’ statements generated from the HWM’s to create some ideas around how to show the agreement to the user.

To start, the statement ‘We could provide a filterable table so a user can choose to view by start date, expiry date, action name, code, etc.’ was used.

It was thought that this was a lot of information and not that easy to differentiate the newly added actions, so another mock-up was made that still used the table idea but separated the new actions into their own area.

Another statement we had was ‘we could arrange by groups of actions’. As there was common data across all rows in the table, to try and minimise the duplication of information some mock-ups were created grouping each set of actions.

To get the most out of the research it was decided to strip out as much information as possible, whilst still leaving the necessary data to understand what was in the agreement, to learn what the users actually needed to see.

We finalised which ‘we could...’ statements to use and agreed on the final design of the agreements page. The content and designs for the email that is sent to the user to trigger them to check and accept the agreement was created; as were the pages outlining the legal information and the confirmation screen letting the user know what will happen next.

What was designed

Next, we put together the hypothesis to take into testing.

Hypotheses

Our tracker for the HMW statements and hypotheses, and the outcomes, can be found within the tracker section (opens in new tab) of our mural board.

For a comprehensive analysis of the research, the user researchers put together a presentation with the findings (opens in new tab) from round 2.

HMW make it easy for users to see the latest change they've signed up for as part of the work they're doing on their land?

Hypothesis 1.1a
If we differentiate how we display the new actions from existing actions by placing them within a grey box,
Then it will be clear to users which actions have been added to the agreement, and which were already there.
We'll know this works when at least 10 out of 12 users are able to successfully review and accept the update.
Disproven

It wasn't immediately clear for users that other actions already in the same agreement were below the grey box with updated actions. This confused users. As a result, the design was altered mid-testing to introduce headings, and a new version of the hypothesis written.

Hypothesis 1.1b
If we include headings to differentiate between new and existing actions,
Then it will be clear to users which actions have been added to the agreement, and which were already there.
We'll know this works when at least 10 out of 12 users are able to successfully review and accept the update.
Inconclusive

Although the addition of the heading helped some users, it was still not obvious to users that this was one single agreement with multiple actions.

Hypothesis 1.2
If we include each set of actions within their own area,
Then it will be clear to users when the actions were added to the agreement.
We'll know this works when at least 10 out of 12 users are able to successfully identify that actions have different start and end dates.
Disproven

It was not clear to all users that the new actions added to the agreement were shown within the area at the top of the page, with actions that were already on the agreement containing in accordions below.

Hypothesis 1.3
If we display each set of actions by the date they were added or amended,
Then the actions will be clearly organised.
We'll know this works when at least 10 out of 12 participants understand that there are different timeframes.
Disproven

The dates caused confusion to users. Part of this confusion was due to the model of having an agreement of a set period of time, so having multiple time periods didn't fit this.

Hypothesis 1.4
If we hide existing actions in accordions when the user is reviewing the newly added actions,
Then it will be clear which actions need to be reviewed.
We'll know this works when at least 10 out of 12 participants understand that these actions are for information and were previously part of the agreement.
Inconclusive

The accordions caused confusion for users with some thinking they were historic agreements, and others being concerned that if they selected them they would be taken somewhere else and 'get lost'.

HMW make it easy for users to see how the change will affect their monthly payments?

Hypothesis 2.1
If we include the annual payment amount (as is shown in the guidance) for each of the actions,
Then the participant wlll understand how much they will receive for each action each year.
We'll know this works when at least 10 out of 12 participants understand what the 'annual payment' value means and in what context.
Disproven

Although having payment information was considered good, many users wanted to see how the calculations had been made, and that more information was required (for example, the amount of land) so that the totals could be checked.

Hypothesis 2.2
If we highlight the change to payments displaying the existing amount and the proposed new amount,
Then participants will understand the financial impact of the updates they are about to accept or reject.
We'll know this works when at least 10 of 12 participants claim to know their new payment amount.
Inconclusive

Even though the difference was shown, many users wanted to compare the new amount to the previous amount they were receiving.

Hypothesis 2.3
If we use bold font to display how much the user will receive each month,
Then it will be clear to the user what their payment will be.
We'll know this works when at least 10 of 12 participants claim to know their new payment amount.
Proven

All users were drawn to the payment details that were shown.

HMW make it easy for users to accept their updated agreement?

Hypothesis 3.1
If we email users with a deep link to the agreements screen,
Then they'll go straight to the agreements screen and be able to carry out the task easily.
We'll know this works when users complete the task.
Proven

All users were able to click the link and complete the task.

Hypothesis 3.2
If we explain what they're signing up for using simple language,
Then they'll understand what they're agreeing to.
We'll know this works when users complete the task.
Inconclusive

Although users were able to complete the task, they did not all fully understand what they were agreeing to. The annual declaration caused confusion as the concept of a single agreement hadn't been understood.

Hypothesis 3.3
If we use a confirmation screen,
Then they'll be reassured the updates they've requested are now live.
We'll know this works when user understands the agreement has been updated and live.
Inconclusive

There was still confusion around the concept of a single agreement.

Conclusions and next steps

We learnt that the whole journey needs to be included in a research session as the participants did not fully understand what the actions were, and why they had been included, which contributed to the concept of a single agreement not being followed.

Much of the content will need some iteration and the page that is showing the updated agreement, or the offer that the user needs to accept, needs to be reconsidered. Some of the other designs from the ideation session could be iterated upon using the findings from this round of research, including more details regarding the land being entered into an action.
