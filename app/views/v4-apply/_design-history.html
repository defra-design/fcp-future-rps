Future Facing application journey - Version 4 Design history

Hypotheses for reducing friction and application times for farmers using many parcels and actions when applying

The problem

The current SFI process (password: farming) for adding actions to parcels is complex, time consuming and doesn't match a farmer's mental model.
This has resulted in:

  • farmers paying agents to help them apply
  • invalid applications being made
  • some farmers avoiding applying altogether as it is a barrier

Product Owner hypotheses

Our Product Owner hypothesised that it may be possible to reduce completion time for SFI applications which involved large numbers of actions and parcels by exploring:

  • giving the user the choice of choosing either actions, or parcels first.
  • allowing multiple actions to be added to the same parcel at once, and vice versa.
  • enabling bulk actions such as 'select all' and quickly applying (for example) the whole available area to chosen land parcels for actions.

Design approach

Iteration 1 - ideation and formative evaluation with farmers

All of the foundational and ideation work can be found on this Mural board.

Foundational assumptions

We approached the ideation work with the following assumptions:

  • Users have decided which actions to apply to which parcels in advance
  • Users plan by parcel first, then consider actions for that parcel
  • Users prefer to select parcels first and then apply actions second
  • Users would like to add one or more actions to one or more parcels
  • Users are visual workers and think in terms of fields
  • Actions with different ambition levels can be grouped together

Research and data analysis

We reviewed previous research findings and analysed land model data to understand how actions are classified by land use types, land types, and other aspects.

The land model data revealed the different actions within SFI and their relationship to land cover type, land cover class, and land use.

This analysis provided an indication of classifications for finding actions and the scale of certain categories users would encounter when searching for actions. As part of the foundational design research, we examined existing patterns from tools such as 'Find funding for land or farms', including search functionality, categorisations, filters, and jump tags for navigating large lists of items. We also explored different models for applying actions to parcels, including one action to many parcels, one parcel to many actions, and variations of these approaches.

"How might we" statements

  • How might we make it quick and easy to apply one or more actions to one or more parcels?
  • How might we display large lists in a useable way?
  • How might we allow a farmer to choose their preferred action when more than one option is available?
  • How might we enable users to review the actions and parcels they have added?
  • How might we enable users to edit the actions and parcels they have added?

High-level user journeys

We sketched high-level user journeys mapping user actions, such as viewing and selecting actions from a list, alongside the backend processes that would filter options based on existing land data, perform available area calculations, and automate tasks on behalf of the user.

Using swim lane diagrams, we documented the components that would be used on specific pages as users progress through the journeyâ€”from filtering and choosing actions to selecting parcels.

We developed four different user journey variations based on whether users select actions first or parcels first, including single actions to many parcels, many parcels to single actions, and other permutations.

These journeys were mapped at a high level, and the team evaluated the comparative merits and drawbacks of each approach.

Screen design exploration

Building on the high-level user journeys, we sketched screen designs using GOV.UK components on Mural. We explored methods to optimise how users find actions from a long list:

  • Categorisation of actions using headings within a long list
  • Search functionality to improve findability when users know which action they require
  • Jump tags linked to action categories to enable rapid navigation
  • 'Select all' functionality to enable users to quickly select all items on a page
  • Bulk selection capability allowing users to select all parcels from a long list without individual selection

Data requirements exploration

We examined the data items related to actions and parcels to determine what information would help users make informed decisions:

  • Payment rates for actions
  • Total size of the parcel
  • Total available area of the parcel
  • Whether SSSI features were present on the parcel

Bulk actions and compliance considerations

We explored the feasibility of allowing users to apply a particular area measurement to multiple selected parcels simultaneously, rather than entering the value individually for each parcel. For example, if users wished to apply the whole available area to all selected parcels, the system could enable bulk application of this value across all selections.

Research questions to be explored with farmers and RPA colleagues included:

  • Whether this functionality would be practical for farmers' working practices
  • Whether this capability is desirable for farmers when applying actions to multiple parcels
  • Whether this approach is permissible from a policy and compliance perspective
  • Whether RPA processors viewing bulk-applied figures might perceive insufficient due diligence in application completion

Findings from user research

We showed farmers 4 different scenarios, using Mural for lightweight and cheap exploration of UI approaches for quickly applying when using lots of parcels. They were:

  • A single action that uses all the available area across 40 parcels
  • A single action which uses part of the available area for 10 parcels
  • A single action that has multiple levels (high, medium, low) for 50 parcels
  • A single action that has a supplemental action(s) on 20 parcels

Action information and guidance

Users need comprehensive information about actions to make informed decisions and ensure compliance. Research revealed that including action 'product descriptions' is essential so users can quickly review and check what they need to do to comply with action requirements.

Key findings about action information:

  • Action code and name are both needed - users did not notice we had switched the order, putting name first
  • Search functionality tested well and was considered helpful for finding actions
  • Users are looking for action-level guidance to support decision-making, specifically: "What do I need to do for this action?"
  • Links to guidance should be provided alongside action listings
  • Payment rate information is crucial for confident decision-making

Listing actions applied to parcels

When displaying actions applied to parcels, especially when more than one action is applied per parcel, long lists proved acceptable to users despite using more screen space and containing some repetitive information.

Essential elements for the shopping basket include:

  • A summary at the top of the basket
  • A detailed breakdown or itemised list of what's in the basket
  • Information about different types of actions (consent/SSSI/HEFER) and clarification that they will be delivered at different times
  • A running total of the monetary value of selected actions on land parcels - this feature tested well and supports confident decision-making

Lists of parcels

Research indicated that pagination is not necessary - long lists of parcels were not the issue we anticipated they might be.

Information needed for land parcels:

  • Total area
  • Total available area

List functionality findings:

  • De-select function is useful in appropriate lists (such as parcel lists)
  • 'Select all' worked well for these highly digitally skilled and confident users
  • A 'none' option is not needed

Search and filtering for actions

Users found search functionality helpful for finding specific actions. Additional filtering options would further improve the user experience.

Effective filtering methods identified:

  • Land cover type
  • Themes organised by areas of interest (as per the Find funding for land or farms tool and previous testing)
  • Countryside Stewardship categories should be removed
  • Filters need to align with the SFI Reformed policy offer

Entering parcel area measurements

The research revealed important insights about how users prefer to enter area measurements for actions:

  • There is no practical way of bulk-applying percentages or hectarage to large numbers of parcels. Farmers do not work in this way, and a bulk application of a percentage figure of the total available area across multiple parcels would fail RPA compliance checks.
  • This task could be rationalised to a single screen where users can enter all or part of the available area
  • The additional screen asking users to choose between 'all' or 'part' should be removed

Decision-making features

  • A 'clear selection' function is needed on every page, allowing users to exit and start again
  • Users are comfortable using the remove function in the application journey
  • Seeing payment rates throughout the journey supports decision-making
  • Running totals and a breakdown of expected payments are important for confident decision-making

Supplemental actions and quantity inputs

Users need a clear way to choose supplemental actions for base actions and add them to land parcels. The research identified specific requirements for handling quantities:

  • A single screen should handle quantity entry
  • When a quantity is mandatory, display it as a statement rather than an input field
  • When the quantity is not mandatory, provide an editable input field that is not pre-populated

Iteration 2 - iteration of design, prototyping and usability testing with farmers

Based on these findings, we iterated the designs and prototyped them (password: futurerps) so we could conduct a second round of usability testing with farmers.

The second round of testing focused on the following scenarios:

  1. Farmer/agent chooses Legume fallow: CNUM3, and applies it to 66 parcels. They want to apply this action to a specific amount of each parcel (this action can be applied to all, or part, of a parcel). (This will test whether the user understands how to add part of the available area to an action)
  2. From review page, go to add another action: Choose 'Manage grassland to reduce nutrient levels in groundwater: WBD8' and apply it to 83 parcels (this will take user to the basket, so we can test their understanding of that flow)
  3. From basket go to add another action: Choose 'Herbal leys: CSAM3' (a total or part area), apply it to 22 parcels, and ask to bulk add total available area to all eligible land parcels. Continue through to basket. (this will test ability to add many actions to many land parcels in bulk)
  4. From review page, new task: they have made a mistake and instead they now want the only original action Legume fallow: CNUM3 - this time added to all the available area of the original 10 parcels. (this will test whether they can clear their basket, return to actions page and go through the flow of adding action to all available area of the parcels)

Farmer's mental model

One of the most significant findings from the second round of testing was that asking farmers to select actions before choosing a parcel does not align with their mental model of their farm and how they practically plan for SFI actions.

One farmer described their planning process as standing in a field with a clipboard, looking around and planning the actions for that particular field, then walking to a neighbouring field and repeating the process. Presenting a large list of actions, followed by a large list of parcels, and then separate screens for entering areas was relatively disorienting for farmers and did not match their real-world workflow. For farmers, the parcel is the primary object of interest when planning their application.

Actions screen

On the actions page, users were able to use the search function effectively. They could also use the jump tags in the content list to navigate to different categories of actions, although some farmers were uncertain about which actions belonged to each category.

Key observations:

  • Some farmers used Control+F to find actions on the page despite having a search function - this is legacy behaviour which they find reliable
  • Some farmers did not notice the search box
  • We have looked at patterns from other services like the Digital Marketplace on GOV.UK that use a wider search box spanning multiple columns, which we may trial in future iterations

Parcels screen

Farmers were able to use 'select all' to select all parcels on the page and did not have issues with long lists, although we recognise this may not be optimal when we combine screens and add mapping functionality later.

Helpful information displayed:

  • Total size of the parcel
  • Total available area of the parcel
  • Triple SI consent indicators
  • Hint text on the parcels

Entering areas screen

On the area screens, where users selected their parcels and then moved to a new screen to enter quantities for those individual parcels, there was a mixed response to the ability to select all parcels and use a button to bulk-apply the total available area to all of them.

Key findings:

  • Some farmers said they would not use this bulk functionality
  • Other farmers could see potential use cases for this feature
  • Many farmers struggled with the design requiring them to select checkboxes to the left of table rows before clicking the 'enter total available area' button
  • Some farmers clicked checkboxes before entering a quantity in the input box, believing this was necessary when it was not
  • Most farmers were able to use the 'reset form' link to clear all fields on the page

Planned improvements:

  • Remove checkboxes from each table row and replace with a button that says 'enter total available area'
  • Iterate the language used - 'reset' may not be the most appropriate phrase

Review screen

On the review page, farmers were generally able to understand the information presented. However, they found it somewhat disorienting to have the action as the primary element with parcel information as the child element within the card. This structure conflicts with their mental model of how they use their farm.

Adding another action

Users were able to choose the correct option to add another action to their parcels. However, some farmers did not notice in the second task that the action could only be applied to the whole available area of each parcel, meaning no areas needed to be manually entered by the user.

We will explore ways to ensure users understand the particular rules for each action.

Hypothesis testing conclusions

The hypothesis that users could add an action to an application, apply it to multiple parcels, and add areas to all those parcels in a quick, bulk manner was partially validated:

  • Selecting multiple parcels at once worked well for users
  • Entering part or percentage areas for parcels in a bulk manner was not validated
  • Entering the total available area for parcels in a bulk manner is conclusive. We will iterate the design to further assess whether this is a User Need for it.

Next steps

Overall, providing actions before parcels conflicts with users' mental model of how they plan and perceive their farm. As a result, we will be switching that order.

We saw strong evidence for combining screens so users can select a parcel, see the actions available for that parcel based on the land data we hold, and enter amounts for those actions as much as possible within the same task flow on the same page. Users could then move to a nearby parcel (as in real life) to repeat the process.

This is a concept we will be exploring next as we integrate mapping visualisations into this work.
