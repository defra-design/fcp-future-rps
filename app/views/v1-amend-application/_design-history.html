Amend an application Design history

Background

One of the initiatives (FDG-74) was to allow a user to re-access a previously submitted application in order to make amendments.

The original flow for the service was allowing a user to apply for a Farm payments technical test (FPTT) agreement. This is achieved via a journey allowing a user to select a parcel, add the required actions, and repeat, until all parcels and actions have been added to the application. This would then be submitted to the Rural Payments Agency (RPA), who would perform their checks, before the sending out an agreement offer. The user would then review this offer, and accept it to form an agreement.

At any point between the user submitting an application and accepting an agreement offer, there should be the capability to allow the user to make any changes to the parcels and/or actions that were submitted. This process involves the RPA 'reopening' the application and sending a new link to the user, in order for them to re-access the application.

That is what was ideated and designed for this version of the prototype. In this scenario:

  • a user has submitted an application, which includes a parcel and action requiring SSSI consent
  • the RPA have not started reviewing the application (so it is not offer stage)
  • the user has decided they no longer want to obtain SSSI consent, so wish to remove the parcel which requires consent

Design approach

The scope for the first iteration was only to allow changes to the parcels and actions that have been included in the application. The ability to change any of the personal or business details was deemed out of scope at this stage.

This lead to us coming up with 3 'how might we' statements to help us ideate the design.

‘How might we’ (HMW) statements

  1. HMW let the user initiate making a change to their application
  2. HMW change the land parcels and/or actions within the application
  3. HMW manage changes that need to be made but are outside our scope

Ideation

The ideation work was carried out on the changing an application/agreement ideation mural board.

We started by putting together a high level flow considering what the users journey would be. It was identified that there were several points in the current service when a user may request to change their application. Adding screen thumbnails to the flow highlighted that some new screens might be required.

When considering how we might let the user initiate making a change, we looked at the three pages that this journey could be started from, and came up with some 'we would...' statements. The three we thought of were:

  • we could add copy to the page informing the user how to amend the information
  • we could add another heading with the text underneath
  • we could add a call to action to the pages

After ideating some flat screen designs it was decided that using a button for the first iteration would involve to much extra technical work, and adding only text could be easily missed. So we came up with a design to add in a heading as well a text.

The main flow of amending any parcels or actions in the application would use the same pages already in the service. It was decided that there would need to be an initial screen that would contain all the information contained in the application, like a check your answers style page, from which the user would then be able to make amendments.

What was designed

An iterated confirmation screen with a new heading and information regarding how to amend an application.

A new page to view an application after it has been submitted.

A new email informing the user that their application is available to amend.

A version of the 'View application' screen with links to allow changing the parcels and actions included.

Conclusions and next steps

The usability testing went well with all the participants being able to carry out the task of removing the parcel that needs SSSI consent.

Moving forward there are a few tweaks that need to be made to the hierarchy of information when viewing the application, as well as reviewing the copy used to inform the users about the SSSI consent.

As a result of technical discussions about implementing the design, there are some changes that need to be made to the flow and some content in the screens. The revised version for this first iteration is in Version 2 of amending an application.
