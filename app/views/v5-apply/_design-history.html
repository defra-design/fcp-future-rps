Future Facing application journey - V5: Maps Design history

Mapping exploration

High level problem

We've seen from existing SFI research that the current mapping solution (password: farming) can be problematic for farmers when applying. They can struggle to:

  • confidently recognise and select the correct land parcel
  • confidently select actions which are eligible to apply to that land parcel
  • understand the implications of selecting those actions and know what they need to do in order to comply

This is important to minimise the risk that we generate failure demand when users realise they have selected the wrong parcel, after they have accepted agreements and contact RPA to request changes.

Deeper analysis - pain points

We analysed mapping findings from previous SFI research and including these in our design thinking:

  • SFI and Mid Tier - if you want to use the map, but if you have application with several smaller parcels, or land spread out, can click on one field, go to the next field, you go back to the original screen, have to navigate back to it, zoom back in, can't navigate around quickly.
  • when you click on the field parcel on the map and it changes colour when you add an option to it. When you go to click on another field, the map shrinks down again, and you have to zoom in again.
  • Users know their parcels by name, not parcel number. It would be useful if users could name their parcels for the purpose of the application process.
  • User described how when thinking about which actions they want to select, they work through the different parcels in that farm and think about the actions they want for that particular parcel.
  • When applying for funding, it would be easier to select actions on a parcel-by-parcel basis (currently, the user is presented with the action selection screen before seeing theit land parcels)

Also, when researching Version 4 of our own application journey ('many parcels, many actons'), we saw strong evidence for combining screens so users can select a parcel, see the actions available for that parcel based on the land data we hold, and enter amounts for those actions as much as possible within the same task flow on the same page. Users could then move to a nearby parcel (as in real life) to repeat the process.

Indeed, one farmer described their planning process as 'standing in a field with a clipboard, looking around and planning the actions for that particular field, then walking to a neighbouring field and repeating the process'.

Design hypothesis

For using a map to apply for SFI funding,
If we match a farmer's physical behaviour (when planning) of standing in a field, noting down actions, checking this, and then moving to the next field]
then farmers will be able to intuitively add parcels, actions, and areas to their application with condidence
We'll know this works/is true when,
we observe farmers quickly find and select particular parcels (and available actions) without errors or excessive navigation while being able to articulate what theyâ€™ve just added.

Design approach

Iteration 1

'How might we...' statements for maps

Drawing from the above pain points and findings, we created the following statements for early mapping design exploration. These speak to both page layout and spefific components (and how these components might interact with each other).

  • How might we help farmers quickly move from parcel to parcel, and farm to farm (if appropriate) while easily understanding their orientation within their holding(s)?
  • How might we help farmers easily dentify their parcels on a map (including just by parcel shape and location on the holding), and confirm they have selected the correct parcel?
  • How might we help farmers understand what actions from existing agreements are present on each and every parcel of their farm(s) so that they're confident the map (and therefore their application) is accurrate and they understand why certain actions may not be available?
  • How might we help farmers understand that available actions for a given parcel are now automatically presented to them on parcel selection based on their land data (hence speeding up their application by removing the 'actions selection' screen before the map on the current SFI service)?
  • How might we speed up applications by changing the trigger for, and language of, error messaging when a farmer is entering hectarage on a selected action?

'How might we...' statements for SFI actions

Also drawing from the pain points and findings mentioned above, we created the following statements for SFI actions within the context of interating with a map to apply for funding.

  • How might we help farmers quickly find an action once available actions have been presented, especially from a long list?
  • How might we help farmers understand what that action is asking of them without cognitive overload within the page or replicating content away from a single source of truth (eg gov.uk guidance)?
  • How might we help farmers understand what that action will pay them per hectare and in total (if used on a parcel)
  • How might we help farmers understand which actions are compatibility within a parcel, and which are not?
  • How might we help farmers avoid 'compatibility errors' for actions (ie choosing actions which cannot be combined within a parcel of land)?

Interactive mapping prototype

We decided that an interactive map prototype would be the most effective and efficient way to explore mapping design.
This was because our work during Version 4 (see Design History) had clearly articulated the mental model of how farmer relates to, and physically interacts with, their parcels of land when planning which SFI actions to do.

We also knew from previous SFI research (see above) that farmers think about their parcels first, then the actions they want to do on those parcels second. Therefore, an interactive map would allow us to explore the interplay between choosing a parcel, being presented with available actions (based on land data we hold), and entering areas for those actions - and then repeating this process for another parcel - all within the same task flow on the same page.

A key aspect of this exploration was to test the design of an 'answers summary' feature within the same page that would help farmers quickly understand which parcels, actions and areas they had chosen, as well as the individual and total annual payments for these choices.
Based on evidence from Version 4 (the farmer walking around his field with his clipboard), we added this to the same screen as the map, as opposed to offsetting it to a Check your Answers screen exclusively.
This is based on the poor usability and frustration we demonstrated when farmers had to 'pogo-stick' between screens to check their answers and then edit their choices.

Designing around this model using an interactive and realistic prototype allowed for strict, task-based scenario testing in order to measure task completion, time to complete tasks and other metrics.

To achieve the necessary interactivity, we used Leaflet.js for the base of the map itself, an open-source JavaScript library which is mobile-friendly.

We then layered on our own code for drawing and selecting parcels, presenting actions on parcel selection, area entry and error messaging and so on.

Findings from user research

We asked farmers to complete 3 scenarios which involved:

  • selecting one of two farms
  • selecting a parcel on the map
  • finding existing actions on the parcel
  • then selecting an action from the list available for that parcel
  • entering an area for that action
  • checking what they'd just done in the 'answers summary' section
  • repeating the process for the next parcel

Scenario testing

The majority of the six participants completed all three scenarios, unaided and quickly.

Hypothesis testing result

The interplay between choosing a parcel, being shown available actions for that parcel from the land data, entering an area for that action, and then viewing the information just entered in the summary in the answers summary below, worked well as the users cycled quickly and without error through these elements over the three tasks.

On the first scenario, Some farmers initially struggled to notice the answers summary section towards the bottom of the page. As they undertook scenario two and three, they became more familiar with the design and cycled round the elements from parcel to action to save to viewing the summary, to another parcel again.

So overall, the core concept of modelling how a farmer plans in their field by walking from parcel to parcel, noting down actions on a clipboard, going to the next parcel to repeat the process - that modelling seemed to work well for the farmers. So the first version seemed to match their mental model.

Next steps - elements to improve

The following elements were observed to need usability improvements, and will be addressed in the next version of the mapping prototype.

Farm selection

When the farmer clicked on one of the farms, the other farm doesn't disappear from the view, so although the map takes them to that farm, the name of the other farm with its details still appears next to the map.

This was a source of confusion for farmers, and so we plan to remove this for the next version.

Parcel names and IDs

Farmers wanted to see parcel IDs as well as parcel names.

This was for accuracy purposes.

Some farmers mentioned that some of their fields actually had the same name, but were different parcels and so the parcel ID provided an extra level of detail and accuracy.

This will be added in the next version.

List of parcels below map

On the details component, which allows the farmer to view more parcels within their parcels list, the number (which is five in this case) of parcels actually confused the farmer.

They were asking "why five more?".

In this case, that was because there were only five more parcels, but they found this confusing.

So we decided we would take the number out for the next version of the design so that the details link would just say "show more parcels".

Findability of actions from previous agreements on selected parcels

We notice farmers vertically scrolling more than desirable to find actions from a previous agreement on a particular parcel, especially when they were looking at the map.

Across the research sessions, we realised that the map was the central element of the page which the farmers orientated around, so this hinted at including information about the existing actions within the popup tooltip on a selected parcel.

The tooltip is too small to include all of this information (or would hide neighbouring parcels if too much content was added to it, affecting map usability).

For the next version, we'll need to consider how we might send the user to the full information while indicating on the map that there are existing actions on a parcel when they click on it.

Answers summary and actions from previous agreements section

Farmers seemed a little bit unsure about the two different sections within the accordion, which holds information about parcels with actions from previous agreements and parcels where the users just added actions to them within this session.

Looking at this component, we can see that the visual hierarchy isn't quite right.

The elements within the sections draw the user's eye and dwarf the actual headings for the sections themselves and so for the next version, we'll address the visual hierarchy so that the section headings are bigger than the content within them.
